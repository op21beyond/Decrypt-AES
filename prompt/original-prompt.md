이번에 너가 할일은 사양체크리스트 문서 작성은 아니고 칩업체의 IP개발자가 설계할 AES decryption hardware 엔진의 개발사양서 문서야.
이 IP는 memory to memory 방식으로 동작하고 버스 마스터 및 례지스터 읽기 쓰기를 위한 슬레이브 인터페이스를 갖고 있어. 인터럽트 출력도 있어. 버스 인터페이스는 IP개발자가 결정하면 돼.
SW가 메모리의 명령 버퍼(circular buffer)에 적어주는 유효한 Descriptor를 읽어서  명령을 처리하고 처리완료하면 디스크립트의 상태 필드에 완료라고 쓰고 디스크립트의 valid 플래그도 클리어해. 해당 디스크립터에 인터럽트를 띄워야하는지를 나타내는 필드가 1세팅되어 있다면 인터럽트를 띄우고 SW가 핸들링완료후 resume 레지스터를 1로 써줄때까지 대기해. Resume이 1로 써지면 자동 클리어하고 명령 버퍼의 다음 디스크립트를 읽어와서 처리를 해. 디스크립트에 인터럽트  발생 지시 플래그가 0이었다면 완료후에 인터럽트도 띄우지않고  바로 다음 디스크립트를 읽어와서 쳐리해. 디스크립트에 last 플래그가 1이었다면 다음 디스크립트를 실행하지 않고 대기해야해. 디스크립트를 읽어왔는데 valid 플래그가 1이 아니라면 inyerval레지스터에 적힌 사이클만큼 후에 다시 디스크립터를 읽어오 재시도를해. 디스크립터에는 입력 데이터의 버퍼ID와  시작 데이터의 주소(버퍼내 오프셋)와 출력데이터의 버퍼ID와 출력 데이터의  시작 주소(버퍼내  오프셋)과 입력 데이터 크기(바이트수), 그중에 마지막 패딩 데이터를 제외한 유효 데이터의 크기, 유효한 출력데이터의 크기와 패딩크기(0일 수 있음)가 적혀있어. 그리고 위에서 말한 valid, 상태, 인터럽트 발생지시, last등의 내용이 헤더에 먼저 오고 그다음에 AES처리할 데이터들이 따라와. 헤더는 32bit aligned.
AES는 128bit CTR체인모드로 decrypt해야해. AES 헤더(IV와 AES CTR 표준 decrypt에 필요한 데이터. 뭐가있지??)가 먼저오고 encrypted data와 패딩이 이어서 와.
