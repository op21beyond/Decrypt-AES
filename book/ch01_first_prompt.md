## 시작은 메모 한 장이었다

모든 설계는 누군가의 머릿속에서 시작된다. 처음엔 완전하지 않아도 된다.  
대개는 화이트보드 낙서이거나, 메신저에 급히 적어 둔 문장이거나, "내일 아침엔 이해하겠지"라는 희망이 담긴 메모다.

이 프로젝트도 마찬가지였다. 시작은 정리되지 않은 요구사항, 메모에 가까운 텍스트였다.  
그러니까 말하자면, 엔지니어링의 출발점치고는 꽤 전형적이었다. 멋진 아키텍처 다이어그램 대신, 약간 급한 마음이 먼저 있었다.

AES는 128bit CTR체인모드로 decrypt해야해. AES 헤더(IV와 AES CTR 표준

throughput 성능 요구도 사양서에 기술되어야해.

## 이 프롬프트가 담고 있는 것

프롬프트를 분해해보면, 다음 정보들이 섞여 있다.

| 담긴 정보 | 구체성 |
|---|---|
| IP 종류 (AES decrypt, memory-to-memory) | 명확함 |
| 동작 방식 (circular buffer, descriptor) | 명확함 |
| 인터럽트 동작 (PAUSE/RESUME 흐름) | 비교적 명확함 |
| 버스 인터페이스 | **"개발자가 결정"** — 미정 |
| 성능 목표 (throughput) | **수치 없음** |
| CRC, 패딩, AES 헤더 상세 | **불완전** |

즉, **개념은 있지만 디테일이 빠진** 상태다.  
보통 이런 문서는 회의 한 번, 메일 두 번, 그리고 "이건 누가 결정하죠?"라는 질문 세 번 정도를 더 거쳐야 비로소 사양서처럼 보이기 시작한다.

일반적인 경우라면 이 상태에서 설계를 시작하면 나중에 많은 것을 되짚어야 한다.  
AI와의 협업에서 흥미로운 지점은 여기다. AI는 빈칸을 억지로 메우거나, 최소한 "여긴 아직 비어 있습니다"라고 표시라도 해 준다.  
사람이 해야 할 일은 그 제안을 평가하고, 틀린 자신감을 제때 잡아주는 것이다.

## 좋은 첫 프롬프트의 조건

이 프롬프트가 효과적이었던 이유를 정리하면 다음과 같다.

### 1. 목적이 명확했다
"사양체크리스트가 아니라 구현 사양서"라고 구체적으로 말했다.  
AI는 문서의 성격을 즉시 파악했다.

### 2. 동작 흐름을 글로 설명했다
비록 완전하지 않았지만, descriptor의 valid 체크 → 처리 → 인터럽트 → PAUSE → RESUME 흐름을 서술했다. AI는 이 흐름을 상태 머신으로 변환했다.

### 3. 빈 칸을 명시적으로 허용했다
"버스 인터페이스는 IP개발자가 결정하면 돼"라고 했다.  
이 표현이 AI에게 결정권을 명시적으로 넘겨준 것이다. 애매하게 두지 않았다.

### 4. 출력 형식을 요청했다
"markdown으로 작성해줘"라고 구체적으로 요청했다.  
Word와 Markdown을 동시에 요청했지만, 실제로는 Markdown만 생성했고 이후 수정하지 않았다.

## 처음부터 완벽한 프롬프트는 필요 없다

하드웨어 엔지니어는 완벽한 사양이 나올 때까지 설계를 시작하지 않는다는 훈련을 받아왔다.  
그 태도는 대부분의 경우 옳다. 실제 칩은 "대충"이라는 단어를 별로 좋아하지 않는다.

바이브코딩에서는 이 습관을 조금 내려놓아야 한다.  
정확히는 완벽주의를 버리는 것이 아니라, 초안 작성의 첫 번째 삽을 AI에게 맡기는 쪽에 가깝다.

> **"70%의 아이디어를 던지고, 나머지 30%는 AI와 함께 채워가라."**

AI는 빠진 부분을 물어보거나, 합리적인 가정을 세워 진행한다.  
가끔은 너무 당당하게 가정하기도 하지만, 그조차도 빈 화면을 바라보는 것보다는 생산적이다.

첫 프롬프트의 역할은 **방향을 정하는 것**이지, 모든 것을 정의하는 것이 아니다.  
첫 공을 너무 완벽하게 던지려 하기보다, 스트라이크존 안으로만 넣자는 마음가짐이 오히려 잘 통한다.

## 프롬프트를 던지는 두 가지 방식

실전에서는 두 가지 스타일이 존재한다.

**스타일 A: 개방형**
```
AES decrypt IP를 만들어야 해. Memory-to-memory 방식이고
Descriptor 기반으로 동작해. 사양서를 작성해줘.
```
→ AI가 많은 것을 결정한다. 빠르지만 의도와 다를 수 있다.

**스타일 B: 반구조형 (이 책에서 사용)**
```
[핵심 동작 설명 + 결정된 사항 나열 + 열린 부분 명시]
빠진 부분은 합리적인 선택으로 채워줘.
```
→ AI가 채울 여지를 주면서도 방향은 잡아둔다. 하드웨어처럼 제약이 많은 도메인에 적합하다.

이 책에서는 일관되게 **스타일 B**를 사용했다.
