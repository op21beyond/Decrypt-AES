# 에필로그 — AI 동료와 일한다는 것 {.unnumbered}

---

## 3일의 기록을 돌아보며

```
4월 12일: original-prompt.md → 사양서 → 전체 RTL (14개 파일)
4월 13일: 호스트 소프트웨어 → 시뮬레이션 테스트벤치 → 버그 수정 시작
4월 14일: 추가 버그 수정 → Verilator 테스트벤치 → GitHub Actions CI
```

사양서, RTL, C 드라이버, 테스트벤치, 합성 스크립트, CI 파이프라인.  
IP 개발의 전체 사이클이 3일 안에 돌아갔다.

문장만 보면 약간 과장처럼 들릴 수 있지만, 실제 체감은 오히려 더 이상했다.  
평소라면 각각 다른 일정표에 있던 일들이 한 주 안에서 아니라, 하루 단위로 서로 붙기 시작했기 때문이다.

혼자였다면 몇 주가 걸렸을 것이다. 팀이 있었다면 더 빠를 수도 있었겠지만, 코드 리뷰와 인터페이스 합의에서 커뮤니케이션 비용이 발생한다.  
AI와의 작업에는 그 비용이 거의 없다. 대신 작성 시간은 줄고 판단 시간은 늘어난다. 하드웨어 엔지니어의 무게중심이 조금 이동한다.

## 바이브코딩이 잘 맞는 영역과 그렇지 않은 영역

**잘 맞는 영역:**

- **표준 프로토콜 구현** — AXI, APB, AHB 같은 스펙이 명확한 인터페이스. AI가 프로토콜을 이미 학습했기 때문에 세부 구현이 빠르다.
- **반복적인 구조** — 레지스터 파일, FIFO, 상태 머신. 패턴이 명확해서 AI가 잘 따라온다.
- **레퍼런스 구현** — 알고리즘의 SW 구현. AES, CRC 같은 표준 알고리즘은 완성도가 높다.
- **인프라 코드** — 빌드 스크립트, CI 설정, 문서 템플릿.

**주의가 필요한 영역:**

- **PDK 특화 설계** — 특정 공정의 타이밍 특성, 셀 선택. AI는 일반적인 원칙만 알고 있다.
- **아날로그·혼합 신호** — Verilog-A, 아날로그 시뮬레이션. AI가 직접 다루기 어렵다.
- **최적화 후반부** — PPA(Power-Performance-Area) 세밀한 조정은 결국 사람의 판단이 필요하다.
- **도메인 특수 지식** — 프로토콜이 공개되지 않은 독점 인터페이스, 내부 팀만 아는 규약.

## 엔지니어의 역할이 바뀐다

바이브코딩 이후, 하드웨어 엔지니어의 역할 중심이 이동한다.

**Before:**  
*"어떻게 구현하는가"* 에 시간의 70%를 쓴다.

**After:**  
*"무엇을 만들 것인가"*, *"올바르게 만들어졌는가"* 에 시간을 더 쓴다.

이것은 역할의 격하가 아니다. 더 높은 수준의 판단이 요구된다.  
AI가 만든 코드를 검토하고, 방향이 잘못됐을 때 수정하고, 엣지 케이스를 발굴하는 능력, 이것이 오히려 더 중요해진다.  
예전에는 손이 먼저 바빴다면, 이제는 판단이 먼저 바빠진다.

도메인 지식이 없는 사람은 AI가 만든 코드가 옳은지 그른지 판단할 수 없다.  
바이브코딩은 도메인 지식을 대체하지 않는다. **도메인 지식의 레버리지를 극대화**한다.  
전문성이 없는 사람이 갑자기 칩을 찍어내게 되는 이야기가 아니라, 전문성이 있는 사람이 훨씬 넓은 범위를 다룰 수 있게 되는 이야기다.

---

## 이 방식으로 개발을 시작하려는 분께

**첫 번째 단계:** 작은 모듈 하나로 시작하라.  
처음부터 전체 IP를 바이브코딩하려 하지 말고, FIFO 하나, 레지스터 파일 하나부터 시작하라.  
AI가 어떻게 동작하는지 체감한 뒤 범위를 넓혀가는 것이 안전하다.

**두 번째 단계:** 컨텍스트 파일을 만들어라.  
`project-instructions.md` 같은 파일을 만들어서 확정된 결정 사항을 기록하라.  
AI는 이 파일을 기준으로 일관된 결과물을 만든다.

**세 번째 단계:** 검증 환경을 일찍 만들어라.  
RTL 설계와 테스트벤치를 동시에 진행하라. AI는 두 작업을 동시에 할 수 있다.  
검증이 없으면 AI의 출력물이 옳은지 확인할 방법이 없다.

**네 번째 단계:** 다른 AI의 리뷰를 받아라.  
같은 AI에게 자신의 코드를 리뷰 요청하는 것은 효과가 제한적이다.  
독립적인 시각을 가진 다른 AI를 리뷰어로 투입하면 놓쳤던 문제를 발견할 수 있다.

## 마지막으로

이 책은 "AI한테 칩 설계를 시켜봤더니 됐다"는 이야기다.  
동시에, "됐지만 그냥 되는 게 아니었다"는 이야기이기도 하다.  
AI는 유능한 신입이자 지치지 않는 실무자처럼 보일 때가 많지만, 여전히 옆자리의 시니어가 필요하다.

방향을 잡고, 구조를 세우고, 결과를 검토하고, 버그를 분석하고, 다른 AI를 리뷰어로 부르는 과정 — 이 모든 것이 엔지니어의 몫이었다. AI는 빠르게 실행했지만, 판단은 사람이 했다.

앞으로 AI의 능력은 더 빠르게 발전할 것이다. 이 책에서 다룬 방식도 1~2년 안에 더 나은 방법으로 대체될 것이다.  
그래서 더 중요한 것은 특정 툴의 사용법보다 협업의 원리다. 어떤 도구가 오더라도, 결국 남는 것은 질문하는 방식과 검증하는 태도다.

중요한 것은, **하드웨어 엔지니어가 이 변화의 주도권을 갖는 것**이다.  
AI를 두려워하거나 무시하는 대신, 어떻게 협업할 것인지를 배우는 쪽이 훨씬 유리하다.  
도구는 계속 바뀌겠지만, 주도권은 여전히 문제를 이해하는 사람 쪽에 남는다.

이 책이 그 첫 걸음에 도움이 되었기를 바란다.
